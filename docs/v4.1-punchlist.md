# V4.1 Punchlist

Working list of features targeted for the next minor release. Order is
priority-ish; subject to change as we learn more.

---

## 1. Goal tracking at the task level *(most wanted)*

**Intent:** every task carries a written goal (and maybe sub-goals or
acceptance criteria). The agent and the human can both see what
"done" looks like for that task. Drift detection becomes possible.

**Open questions**
- Where does the goal live? Document store next to the existing task
  fields? Separate `goals` entity that links to a task? Multiple
  sub-goals per task or a single freeform string?
- Who writes the goal — human, agent, or both? Edit lifecycle.
- Does the canvas surface the goal in the session widget header, in a
  dedicated panel, or only on hover/click?
- Status: open / in-progress / done / abandoned? Auto-derive from agent
  recap signals or explicit toggle?
- Telemetry: do we want "% of tasks with explicit goals" /
  "% with goals achieved per session" as a metric?

**Sketch of scope**
- New `Goal` entity (or `task.goal: string | null` field) in `docStore`.
- Server: CRUD endpoints + SSE deltas.
- Frontend: rich editor on a task card; list filter "tasks without
  goals"; goal recap in the session header.

**Why most-wanted:** the agent loop can hum for hours without anyone
checking whether it's still pointed at the right thing. Explicit goals
give the human a one-glance correctness check.

## 2. Dispatcher — LLM-assisted Tinstar admin

**Intent:** a chat-style entry point that drives Tinstar's control
plane. "Spawn three reviewers on the auth refactor task and tell me
when they all reach idle." "Move the OTA epic into the cmsandbox
initiative." "Rename every breakout room with no traffic in 24h."

**Open questions**
- Does the dispatcher live in the Tauri shell, in the canvas as a
  widget, or as a top-level "command palette" overlay?
- Does it use Tinstar's own session backend (it IS a session) or a
  dedicated Claude API call with tool access to the Tinstar API?
- Tool surface: full `/api/*` or curated wrappers? Where do we draw the
  line between "convenience" and "agent loop"?
- Authorization model — does it need user-confirm for destructive ops?
- Memory / context — does the dispatcher carry a persistent backlog
  across sessions, or is each conversation isolated?

**Sketch of scope**
- New session adapter that wraps a Claude API loop with Tinstar tools.
- React `<DispatcherChat />` component (probably full-height side
  panel, hotkey-toggleable).
- Tool catalog generated from `/api/openapi.json` plus hand-curated
  high-leverage actions.

**Why it matters:** scaling Tinstar past ~20 sessions becomes a UX
problem long before it becomes a perf problem. A natural-language
admin layer is the way out.

## 3. Better Tauri onboarding — first-run server + workspace setup

**Intent:** the V4.0 desktop app currently has a hardcoded `infrapoc:5273`
backend at build time. First launch should walk a new user through:
"Where is your tinstar backend? Don't have one yet? I'll start one for
you. Once you're connected, let's create your first workspace."

**Open questions**
- Does the Tauri app spawn its own tinstar via the npm CLI shell-out,
  bundle a sidecar, or assume the user has tinstar installed
  separately? (Phase 2c+d in the original Tauri plan.)
- For remote backends — show the existing `TINSTAR_CORS_ORIGINS`
  warning text + the exact command to set on the remote, or have the
  desktop probe + tell the user what's missing?
- First-run space creation — copy the in-app "create space" flow or
  a dedicated wizard with example data?
- How to handle "I already have a backend running" detection?

**Sketch of scope**
- This is largely Phase 2b/c/d of the original Tauri plan, which we
  deferred for V4.0. Pull from
  `docs/superpowers/plans/2026-04-30-tauri-phase-2.md` for detail.
- `<FirstRunSetup />` React component (already specced).
- Tauri commands: `get_config`, `save_config`, `probe_backend`,
  `start_local_backend`, `stop_local_backend`.
- `desktop.json` config in `app_config_dir()`.

**Why it matters:** V4.0 ships a native binary that's currently only
useful to people who already know how to run a tinstar backend. V4.1
should make it work for anyone who downloads the .dmg.

## 4. Inbox view as an alternative to hierarchy view

**Intent:** today the canvas is hierarchical (initiative → epic → task →
session). Some workflows want a flat, time-ordered "inbox" — show me
everything that needs my attention right now, regardless of where it
sits in the tree.

**Open questions**
- What constitutes "needs attention"? Idle agents waiting on me?
  Sessions with new recap entries since last view? Failed sessions?
  All of the above with filters?
- Does the inbox replace the hierarchy view (toggle), live alongside
  (split-pane), or sit in a different space entirely?
- Sort order — most-recent activity, longest-stuck, oldest-unanswered?
  All three with a switcher?
- Does an item disappear from the inbox when "handled" (clicked into,
  responded to, etc.) or stick until explicitly archived?
- Per-task or per-session granularity? Probably per-session, but with
  task context shown on each row.

**Sketch of scope**
- New top-level view component (`<InboxView />`) sibling to the
  current canvas.
- Server: maybe a derived endpoint or just compute from existing
  state. New deltas if we want unread tracking.
- Hotkey + sidebar toggle to switch between hierarchy and inbox.

**Why it matters:** hierarchy is the right organizing structure but
the wrong daily-driver UI when you have dozens of active sessions
across many initiatives. Inbox is the triage view.

## 5. Stop thrashing `.mcp.json` per task

**Intent:** today, every task switch rewrites `.mcp.json` to point the
nats-channel-mcp at that task's role/subject/socket. The file becomes
a per-task source of drift that shows up in `git status` constantly,
gets accidentally committed, and produces noisy diffs like
`quickfeatures → bugfixes` that have nothing to do with the work.

**Open questions**
- Replace per-task mutation with env-var substitution at MCP launch
  (`${TINSTAR_NATS_ROLE}`, `${TINSTAR_NATS_SUBJECT}`,
  `${TINSTAR_NATS_SOCKET}`)? The MCP client needs to support env
  expansion in args — verify before committing to the approach.
- Or: move the per-task config out of `.mcp.json` entirely and into a
  tinstar-managed file under `$TINSTAR_CONFIG_HOME/` that the MCP
  wrapper reads at startup. `.mcp.json` then stays static.
- Either way, what does the existing `.mcp.json` look like after the
  change — empty, or just the static parts?
- Migration: how do we land this without breaking already-running
  agent sessions whose MCP processes are bound to the old shape?

**Sketch of scope**
- Decide: env-var substitution vs. external config file vs. a tiny
  wrapper script.
- Update the spawn path so per-task values flow through the new
  channel instead of editing `.mcp.json`.
- Add `.mcp.json` to a "should be static" invariant — flag in CI or a
  pre-commit check if it diverges.

**Why it matters:** every agent session leaves behind a dirty
working tree purely from infrastructure plumbing. It pollutes commits
(see the `chore(mcp): rename ... quickfeatures -> bugfixes` commit
that landed on V4.1 only because the file happened to be dirty when
something else needed committing). It also makes it impossible to
tell at-a-glance whether a real `.mcp.json` change has occurred.
